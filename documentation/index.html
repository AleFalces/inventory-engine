OmniCore Inventory Engine

Una guía paso a paso de lo que construimos, por qué tomamos cada decisión, y cómo entender el código si es la primera vez que trabajás con Java y Spring Boot. No se asume experiencia previa en Java.

1

¿Qué es este proyecto?

OmniCore Inventory Engine es un sistema backend que gestiona el inventario de productos para múltiples empresas al mismo tiempo. Pensalo como el motor que está detrás de una tienda online: lleva el control de cuántos productos hay en stock, permite agregarlos, modificarlos y consultarlos.

Lo que hace especial a este sistema es que está diseñado para ser usado por muchas empresas distintas a la vez (como un SaaS), manteniendo los datos de cada una completamente separados y seguros.

🏭

Motor de inventario

Gestiona stock de productos en tiempo real con alta concurrencia.

🏢

Multi-empresa (SaaS)

Múltiples clientes usan el mismo sistema, aislados entre sí.

🔒

Seguridad por diseño

La separación de datos es la prioridad número uno.

El stack tecnológico

Si venís de JavaScript/Python, algunos de estos nombres son nuevos. Acá te explicamos qué hace cada uno y por qué lo elegimos.

Tecnología ¿Qué es? ¿Por qué la usamos?
Java 21 El lenguaje de programación. Como Python pero con tipos estrictos. Maduro, rápido, y dominante en backends empresariales.
Spring Boot Un framework para Java. Como Express.js pero para Java. Arranca una app completa con muy poco código de configuración.
PostgreSQL La base de datos. Como MySQL pero más potente. Soporte nativo para concurrencia, tipos avanzados y transacciones robustas.
JPA / Hibernate Una capa que conecta Java con la base de datos. Escribís clases Java y él genera el SQL automáticamente.
Lombok Una librería que genera código repetitivo automáticamente. Evita escribir getters, setters y constructores a mano.
MapStruct Convierte objetos de un tipo a otro automáticamente. Evita copiar campos a mano entre la entidad y el DTO.
Testcontainers Levanta una base de datos real en Docker para los tests. Los tests son 100% fieles a producción. Sin sorpresas.
Spring Security + JWT Framework de seguridad + tokens de autenticación firmados. Autenticación stateless. El tenantId y el role se extraen del token, nunca del cliente.
Bean Validation Anotaciones @NotBlank, @Min, @Size en los DTOs. Inputs inválidos son rechazados con 400 antes de llegar al servicio.
RBAC (Method Security) Roles ADMIN / VIEWER embebidos en el JWT. VIEWER solo puede leer. ADMIN puede escribir. Controlado con @PreAuthorize.

Multi-tenancy: el concepto más importante

Tenant = cliente / empresa. Multi-tenancy significa que el sistema sirve a muchas empresas al mismo tiempo, en una sola instalación.

Imaginá que sos dueño de un edificio con muchos departamentos. Cada familia (tenant) tiene su espacio privado. Comparten el edificio, pero nunca pueden entrar al departamento de otro. Eso es lo que hacemos con los datos.

¿Cómo lo implementamos?

Usamos la estrategia más simple: una sola base de datos, una sola tabla, pero cada fila tiene una columna tenant_id que identifica a qué empresa pertenece. 

-- Tabla products en PostgreSQL
SELECT * FROM products;

 id | tenant_id      | sku      | name          | stock
----+----------------+----------+---------------+-------
  1 | empresa-alpha  | SKU-001  | Widget Pro    |   100  ← empresa A
  2 | empresa-alpha  | SKU-002  | Gadget X      |    50  ← empresa A
  3 | empresa-beta   | SKU-001  | Widget Pro    |   200  ← empresa B (¡mismo SKU, diferente empresa!)
¿Por qué este enfoque?

Alternativas más complejas como una base de datos por tenant o un schema por tenant son más caras de mantener. Con una sola base de datos, una sola migración actualiza a todos los clientes. La seguridad la garantizamos en el código.

Las 3 reglas de oro del aislamiento

1. Toda entidad tiene tenantId

Si falta este campo en una tabla, es un bug crítico de seguridad.

2. Todo query filtra por tenantId

Nunca findAll(). Siempre findAllByTenantId("empresa-alpha"). Si buscás por ID, también: findByTenantIdAndId("empresa-alpha", 1).

3. El tenantId nunca viene del usuario

El servidor lo obtiene del token de autenticación. Un usuario no puede mentir y decir que es de otra empresa.

Regla crítica

Un bug que deje a un cliente ver datos de otro cliente es un incidente de seguridad gravísimo (P0). Esta regla no es negociable en ninguna parte del código.

TDD — Tests primero, código después

TDD (Test Driven Development) es una metodología donde primero escribís el test que falla, y después escribís el código mínimo para que ese test pase. Parece al revés, pero tiene mucho sentido.

🔴 Red

Escribís el test. Falla porque el código no existe todavía. Eso está bien.

🟢 Green

Escribís el código mínimo para que el test pase. Sin agregar de más.

🔵 Refactor

Mejorás el código sin romper los tests. Limpieza sin miedo.

¿Por qué trabajar así?

Los tests son la red de seguridad del proyecto. Si en el futuro alguien cambia un método y rompe algo, los tests lo detectan antes de que llegue a producción. También obligan a diseñar código que sea fácil de testear, lo que suele significar código más simple y limpio.

Testcontainers — Base de datos real en tests

Cuando corrés los tests, Testcontainers levanta automáticamente un contenedor Docker con PostgreSQL, conecta Spring Boot a él, corre todos los tests, y destruye el contenedor al final. Todo sin que tengas que hacer nada. 

// src/test/java/.../TestcontainersConfiguration.java
@Bean
@ServiceConnection  // ← Spring Boot detecta la URL sola, sin config manual
public PostgreSQLContainer postgresContainer() {
    return new PostgreSQLContainer("postgres:latest");
}

¿Por qué no usamos H2 (base de datos en memoria)?

Testear con H2 en lugar de PostgreSQL es como ensayar una cirugía en un maniquí de plástico. Testcontainers usa el "paciente real" en un entorno controlado.

¿Por qué separar en domain y api?

domain es el corazón del sistema: las reglas de negocio, la base de datos, la lógica. No sabe nada de HTTP ni de cómo se ve la respuesta.

api es la fachada: lo que el mundo exterior ve. Transforma los datos del dominio en respuestas HTTP limpias. Si mañana queremos agregar una API GraphQL, solo tocamos api/ sin tocar el dominio.

La entidad Product

Una entidad en JPA es una clase Java que representa una tabla en la base de datos. Cada instancia de la clase es una fila en esa tabla. 

@Entity                          // ← "esta clase es una tabla"
@Table(name = "products")         // ← nombre de la tabla en PostgreSQL
@Getter @Setter                  // ← Lombok genera getters y setters
@NoArgsConstructor               // ← Lombok genera constructor vacío (JPA lo necesita)
@AllArgsConstructor @Builder     // ← Lombok genera constructor completo y patrón Builder
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;              // ← la base de datos asigna el ID automáticamente

    @Column(name = "tenant_id", nullable = false)
    private String tenantId;       // ← a qué empresa pertenece este producto

    @Column(nullable = false)
    private String sku;            // ← código único del producto por empresa

    @Column(nullable = false)
    private String name;           // ← nombre del producto

    private String description;    // ← opcional, puede ser null

    @Column(nullable = false)
    private Integer stock;         // ← unidades disponibles

    @Version
    private Long version;          // ← protección contra escrituras simultáneas
}

¿Qué es @Version?

Imaginá que dos usuarios compran el último producto al mismo tiempo. Sin protección, los dos podrían decrementar el stock y quedarte con un número negativo. @Version implementa bloqueo optimista: cada vez que se modifica una fila, el número de versión sube. Si dos personas intentan modificar la misma fila al mismo tiempo, la segunda recibe un error y debe reintentar. Esto es esencial en un motor de inventario.

¿Qué es Lombok?

En Java "puro", tendrías que escribir a mano docenas de líneas de código repetitivo (getters, setters, constructores). Lombok las genera automáticamente durante la compilación con solo una anotación. El código es más corto y legible, sin perder funcionalidad.

ProductRepository — Hablar con la base de datos

El repositorio es la única parte del código que hace queries a la base de datos. Con Spring Data JPA, en muchos casos solo definís la firma del método y Spring genera el SQL automáticamente. 

public interface ProductRepository extends JpaRepository<Product, Long> {

    // Spring genera: SELECT * FROM products WHERE tenant_id = ?
    List<Product> findAllByTenantId(String tenantId);

    // Spring genera: SELECT * FROM products WHERE tenant_id = ? AND sku = ?
    Optional<Product> findByTenantIdAndSku(String tenantId, String sku);

    // Spring genera: SELECT * FROM products WHERE tenant_id = ? AND id = ?
    // Protege el GET por ID: aunque alguien adivine un ID ajeno, retorna vacío
    Optional<Product> findByTenantIdAndId(String tenantId, Long id);
}
¿Por qué usamos Optional?

Optional<Product> es como decirle a quien llama al método: "puede que encuentre el producto, o puede que no". Obliga al código que lo usa a manejar el caso donde no existe, evitando el clásico error NullPointerException.

Regla de aislamiento

Notá que ningún método permite consultar sin tenantId. Si alguien quisiera agregar un findBySku(String sku), devolvería resultados de todos los tenants — eso está prohibido.

Capa de Servicio — La lógica de negocio

El servicio es el cerebro de la aplicación. Conecta el repositorio (base de datos) con el controller (API). Aquí viven las reglas de negocio: qué está permitido hacer y qué no.

Los DTOs — Records de Java

Un DTO (Data Transfer Object) es el objeto que viaja entre capas. Usamos Java Records porque son inmutables por diseño: una vez creados, no se pueden modificar. Perfecto para datos que solo transportan información. 

// Entrada: lo que recibe el sistema del cliente
public record CreateProductRequest(
    @NotBlank String sku,
    @NotBlank String name,
               String description,
    @NotNull @Min(0) Integer stock
) {}

// Salida: lo que el sistema devuelve al cliente
public record ProductResponse(
    Long id, String tenantId, String sku,
    String name, String description, Integer stock
) {}
¿Por qué separar entrada y salida?

El cliente no debe poder enviar el id ni el tenantId — esos los asigna el servidor. El record de entrada solo tiene los campos que el cliente tiene derecho a definir.

MapStruct — Conversión automática entre objetos

MapStruct genera automáticamente el código que copia campos de un objeto a otro. Sin él, tendrías que escribir esto a mano para cada campo: response.setSku(product.getSku()). MapStruct lo hace en tiempo de compilación, sin reflexión, lo que lo hace muy rápido. 

@Mapper(componentModel = "spring")
public interface ProductMapper {

    ProductResponse toResponse(Product product);   // ← MapStruct genera la implementación

    @Mapping(target = "id",       ignore = true)   // ← el cliente no puede fijar el ID
    @Mapping(target = "tenantId", ignore = true)   // ← el servidor lo asigna
    @Mapping(target = "version",  ignore = true)   // ← lo maneja Hibernate
    Product toEntity(CreateProductRequest request);
}

ProductService — Reglas de negocio

@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)   // ← todas las operaciones son read-only por defecto
public class ProductService {

    @Transactional   // ← sobreescribe: esta operación escribe en la BD
    public ProductResponse createProduct(String tenantId, CreateProductRequest request) {

        // Regla de negocio: no puede haber dos productos con el mismo SKU en el mismo tenant
        productRepository.findByTenantIdAndSku(tenantId, request.sku())
            .ifPresent(p -> { throw new ProductAlreadyExistsException(tenantId, request.sku()); });

        var product = productMapper.toEntity(request);
        product.setTenantId(tenantId);   // ← el servidor, no el cliente, fija el tenant

        return productMapper.toResponse(productRepository.save(product));
    }
}

Autenticación JWT (Fase 8)

Todos los endpoints requieren un token JWT en el header Authorization: Bearer <token>. El servidor extrae el tenantId del subject del token — el cliente nunca lo envía directamente. Sin token o con token inválido: HTTP 401 Unauthorized.

Ejemplo de uso

# Obtener un token (ejemplo con JwtUtil en dev)
TOKEN=eyJhbGciOiJIUzI1NiJ9...

# Crear un producto
curl -X POST http://localhost:8080/api/v1/products \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"sku":"SKU-001","name":"Widget Pro","stock":100}'

# Listar productos (paginado)
curl "http://localhost:8080/api/v1/products?page=0&size=20" \
  -H "Authorization: Bearer $TOKEN"

# Buscar por SKU
curl http://localhost:8080/api/v1/products/SKU-001 \
  -H "Authorization: Bearer $TOKEN"

Manejo de errores — GlobalExceptionHandler

En lugar de manejar errores en cada controller, tenemos un handler centralizado que convierte las excepciones en respuestas HTTP con el código correcto.

Fase 4 — CRUD completo ✓ Completa

Fase completada con 25/25 tests en verde. Se agregaron PUT /{sku} y DELETE /{sku} a las tres capas siguiendo el ciclo TDD estricto.

Lo que aprendimos — el patrón de stubs

Para que los tests compilen y fallen (RED) antes de implementar, usamos stubs mínimos que lanzan UnsupportedOperationException. Esto garantiza que los tests corren en rojo — no que simplemente el proyecto no compila.

Implementación final — updateProduct y deleteProduct

@Transactional
public ProductResponse updateProduct(String tenantId, String sku, UpdateProductRequest request) {
    var product = productRepository.findByTenantIdAndSku(tenantId, sku)
            .orElseThrow(() -> new ProductNotFoundException(tenantId, sku));

    product.setName(request.name());
    product.setDescription(request.description());
    product.setStock(request.stock());

    return productMapper.toResponse(productRepository.save(product));
}

@Transactional
public void deleteProduct(String tenantId, String sku) {
    var product = productRepository.findByTenantIdAndSku(tenantId, sku)
            .orElseThrow(() -> new ProductNotFoundException(tenantId, sku));

    productRepository.delete(product);
}
¿Por qué no se puede cambiar el SKU?

El SKU es la identidad del producto dentro del tenant. Cambiarlo rompería referencias en órdenes de compra, movimientos de stock y cualquier sistema externo integrado. Si un producto necesita un SKU nuevo, se crea uno nuevo y se da de baja el viejo.

Fase 6 — Trazabilidad de stock ✓ Completa

Fase completada con 39/39 tests en verde. Cada ajuste de stock queda registrado como un evento inmutable para auditoría.

Modelo StockMovement

@Entity
public class StockMovement {
    private String  tenantId;     // aislamiento multi-tenant
    private String  productSku;   // SKU del producto afectado
    private Integer delta;        // cantidad ajustada (+ entrada, - salida)
    private String  reason;       // motivo del ajuste
    private Integer stockBefore;  // stock antes del ajuste
    private Integer stockAfter;   // stock después del ajuste
    private Instant createdAt;    // timestamp del evento
}
¿Por qué registrar movimientos?

Sin trazabilidad, si el stock de un producto baja de 100 a 0 no sabés si fueron 100 ventas, un ajuste manual, o una pérdida por merma. El historial de movimientos permite auditar cada cambio, detectar errores y cumplir con requisitos legales de inventario.

Integración con adjustStock

// Cada vez que adjustStock() se llama, también persiste un movimiento
stockMovementRepository.save(StockMovement.builder()
    .tenantId(tenantId).productSku(sku)
    .delta(request.delta()).reason(request.reason())
    .stockBefore(stockBefore).stockAfter(newStock)
    .createdAt(Instant.now())
    .build());

¿Por qué paginar?

Fase 8 — Spring Security + JWT ✓ Completa

Fase completada con 49/49 tests en verde. Se reemplazó el header X-Tenant-ID por autenticación real con tokens JWT firmados.

¿Qué es un JWT?

Un JWT (JSON Web Token) es un string en formato header.payload.signature que el servidor firma con un secreto. El cliente lo incluye en cada request. El servidor verifica la firma — si es válida, extrae el tenantId del subject sin necesidad de consultar la base de datos. 

// Estructura del JWT que genera JwtUtil
{
  "sub": "empresa-alpha",   ← tenantId aquí
  "iat": 1712700000,
  "exp": 1712786400       ← expira en 24 horas
}

Flujo de autenticación

1. Cliente envía request con Authorization: Bearer <token>

El token JWT va en el header HTTP en cada petición.

2. JwtAuthenticationFilter intercepta el request

Extrae el token, llama a JwtUtil.isValid(). Si falla → 401. Si pasa → continúa.

3. El tenantId se guarda en el SecurityContext

Se crea un UsernamePasswordAuthenticationToken con el tenantId como principal.

4. El controller recibe el tenantId automáticamente

@AuthenticationPrincipal String tenantId — Spring lo inyecta desde el contexto de seguridad.

JwtAuthenticationFilter

@Override
protected void doFilterInternal(HttpServletRequest request, ...) {
    String header = request.getHeader("Authorization");
    if (header != null && header.startsWith("Bearer ")) {
        String token = header.substring(7);
        if (jwtUtil.isValid(token)) {
            String tenantId = jwtUtil.extractTenantId(token);
            var auth = new UsernamePasswordAuthenticationToken(tenantId, null, List.of());
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
    }
    chain.doFilter(request, response);   // sin auth → Spring devuelve 401
}
Importante — jwt.secret en producción

El secreto en application.properties es solo para desarrollo. En producción debe ser una variable de entorno con al menos 256 bits de entropía. Nunca commitear el secreto real al repositorio.

Fase 9 — Login endpoint ✓ Completa

Fase completada con 62/62 tests en verde. Hasta este punto los JWT solo existían en los tests. Esta fase expone POST /api/v1/auth/login para que cualquier cliente real pueda obtener un token.

El problema que resuelve

En la Fase 8 implementamos validación de JWT, pero nunca explicamos cómo un cliente obtiene ese token. Era como poner una cerradura sin dar la llave. Esta fase agrega el endpoint que emite tokens a cambio de credenciales válidas.

La entidad Tenant — ¿qué es una "empresa cliente"?

Un Tenant es una empresa que usa nuestro sistema. En la base de datos guardamos su identificador y su contraseña hasheada — nunca la contraseña en texto plano. 

@Entity
@Table(name = "tenants")
public class Tenant {
    @Id @GeneratedValue
    private Long   id;

    @Column(name = "tenant_id", nullable = false, unique = true)
    private String tenantId;      // "empresa-alpha"

    @Column(name = "password_hash", nullable = false)
    private String passwordHash;  // "$2a$10$..." ← BCrypt, nunca texto plano
}
¿Por qué no guardamos la contraseña directamente?

Si la base de datos se filtra, el atacante obtiene los hashes — no las contraseñas reales. BCrypt es un algoritmo de hashing diseñado para ser deliberadamente lento, lo que hace inviable el ataque por fuerza bruta. En otros lenguajes como Node.js usarías bcrypt.hash(); en Spring usamos passwordEncoder.encode() — mismo concepto.

¿Qué hace AuthService.login()?

La lógica de login es simple pero hay que hacerla en el orden correcto:

public LoginResponse login(String tenantId, String password) {

    // 1. Buscar el tenant en la BD — si no existe, lanzamos 401
    //    (NO decimos "tenant no encontrado" para no dar pistas al atacante)
    var tenant = tenantRepository.findByTenantId(tenantId)
            .orElseThrow(InvalidCredentialsException::new);

    // 2. Verificar la contraseña contra el hash guardado en BD
    if (!passwordEncoder.matches(password, tenant.getPasswordHash())) {
        throw new InvalidCredentialsException();  // → HTTP 401
    }

    // 3. Credenciales válidas → generar y devolver el JWT
    return new LoginResponse(jwtUtil.generateToken(tenantId, tenant.getRole()));
}
Seguridad: siempre el mismo mensaje de error

Tanto si el tenant no existe como si la contraseña es incorrecta, devolvemos la misma excepción (InvalidCredentialsException → 401). Si dijésemos "tenant no encontrado" vs "contraseña incorrecta", estaríamos confirmando qué tenantIds existen — un vector de ataque llamado user enumeration.

Flujo completo de una autenticación

1. Cliente: POST /api/v1/auth/login

Body: {"tenantId": "empresa-alpha", "password": "mipassword"}

2. AuthController recibe el request y llama a AuthService.login()

El controller no tiene lógica — delega todo al servicio.

3. AuthServiceImpl busca el tenant y valida con BCrypt

passwordEncoder.matches("mipassword", "$2a$10$...") — compara la contraseña con el hash.

4. JwtUtil.generateToken() crea y firma el token

El JWT incluye tenantId como subject y el role como claim. Expira en 24h.

5. Respuesta: {"token": "eyJhbGci..."}

El cliente guarda este token y lo incluye en cada request siguiente como Authorization: Bearer <token>.

Fase 10 — Registro de tenants ✓ Completa

Fase completada con 68/68 tests en verde. Cierra el ciclo de autenticación: un nuevo cliente puede registrarse y acceder a la API sin intervención manual en la base de datos.

¿Por qué no devolvemos el token al registrarse?

Es una decisión de diseño deliberada. Separar registro de login tiene ventajas: el registro puede requerir validación adicional en el futuro (email de confirmación, aprobación manual, etc.). El cliente sabe que debe hacer dos llamadas: primero /register, después /login. Es más explícito y extensible.

DTOs como Java Records — inmutabilidad by design

En Java usamos Records para los DTOs de request/response. Si venís de Python pensá en dataclass(frozen=True); si venís de TypeScript, pensá en una interface con todos los campos readonly. Una vez creado el record, sus campos no se pueden modificar. 

// Record = clase inmutable con constructor, getters y equals/hashCode automáticos
public record RegisterRequest(
    @NotBlank String     tenantId,   // acceso: request.tenantId()
    @NotBlank @Size(min = 8) String password,
    TenantRole role                   // null → default VIEWER
) {
    // Constructor de conveniencia — si no pasás role, es VIEWER
    public RegisterRequest(String tenantId, String password) {
        this(tenantId, password, TenantRole.VIEWER);
    }
}

Qué hace AuthService.register()

public RegisterResponse register(String tenantId, String password, TenantRole role) {

    // 1. ¿Ya existe? → 409 Conflict
    if (tenantRepository.existsByTenantId(tenantId)) {
        throw new TenantAlreadyExistsException(tenantId);
    }

    // 2. Hashear la contraseña antes de guardar — nunca texto plano
    var tenant = Tenant.builder()
            .tenantId(tenantId)
            .passwordHash(passwordEncoder.encode(password))  // BCrypt
            .role(role)
            .build();

    tenantRepository.save(tenant);                // INSERT en tabla tenants
    return new RegisterResponse(tenantId);         // 201 Created
}

El flujo completo que ahora es posible

# 1. Registrarse
POST /api/v1/auth/register
{"tenantId": "mi-empresa", "password": "mipassword123", "role": "ADMIN"}
→ 201 {"tenantId": "mi-empresa"}

# 2. Obtener token
POST /api/v1/auth/login
{"tenantId": "mi-empresa", "password": "mipassword123"}
→ 200 {"token": "eyJhbGci..."}

# 3. Usar la API
GET /api/v1/products
Authorization: Bearer eyJhbGci...
→ 200 {"content": [], "totalElements": 0, ...}

Fase 11 — Bean Validation ✓ Completa

Fase completada con 73/73 tests en verde. Antes de esta fase era posible registrarse con tenantId: "" o crear un producto con stock -500. Bean Validation lo impide.

¿Por qué validar en el DTO y no en el servicio?

El servicio debería tener lógica de negocio, no de formato. Validar que un string no esté vacío es responsabilidad de la frontera del sistema — donde entra el dato del exterior. Si lo validamos en el DTO con anotaciones, Spring lo rechaza automáticamente con 400 y ni siquiera llega al servicio. Es más limpio y más fácil de testear.

Las anotaciones disponibles

AnotaciónQué validaEjemplo de uso
@NotNullEl campo no puede ser nullcampos Integer, objetos
@NotBlankString no nulo y no vacío (incluye espacios)sku, name, tenantId
@Min(0)Número ≥ al valor indicadostock no puede ser negativo
@Size(min=8)String de al menos N caracterespassword mínimo 8 chars

¿Cómo se activa? — la clave está en @Valid

Las anotaciones en el DTO solas no hacen nada. Necesitás añadir @Valid delante del @RequestBody en el controller para que Spring las procese. 

// Sin @Valid → Spring ignora las anotaciones del DTO
@PostMapping
public LoginResponse login(@RequestBody LoginRequest request) { ... }

// Con @Valid → Spring valida antes de llamar al método
@PostMapping
public LoginResponse login(@Valid @RequestBody LoginRequest request) { ... }

¿Qué pasa cuando falla la validación?

Spring lanza una MethodArgumentNotValidException automáticamente. Nuestro GlobalExceptionHandler la captura y devuelve un 400 con el campo problemático. Este handler ya existía desde la Fase 3 — no fue necesario cambiarlo. 

// En GlobalExceptionHandler — ya existía, captura la excepción de validación
@ExceptionHandler(MethodArgumentNotValidException.class)
@ResponseStatus(HttpStatus.BAD_REQUEST)
public ErrorResponse handleValidation(MethodArgumentNotValidException ex) {
    String message = ex.getBindingResult().getFieldErrors().stream()
            .map(e -> e.getField() + ": " + e.getDefaultMessage())
            .findFirst()
            .orElse("Invalid request");
    return new ErrorResponse(message);  // → {"message": "password: size must be between 8 and ..."}
}
Records y anotaciones de validación

Los Java Records generan su constructor automáticamente a partir de los componentes declarados. Las anotaciones de validación se colocan directamente en los componentes del record — Spring las respeta igual que en una clase normal.

Fase 12 — RBAC: roles por tenant ✓ Completa

Fase completada con 80/80 tests en verde. Introduce los roles ADMIN y VIEWER. Un VIEWER autenticado que intente crear un producto recibe un 403 — está autenticado, pero no autorizado.

Autenticación vs Autorización — la diferencia clave

Autenticación: ¿quién sos? (JWT válido → sabemos que sos "empresa-alpha")
Autorización: ¿qué podés hacer? (¿tenés el rol ADMIN para crear productos?)

Hasta la Fase 11 solo teníamos autenticación. Cualquier tenant autenticado podía hacer cualquier cosa. RBAC (Role-Based Access Control) agrega la segunda capa.

Paso 1 — El enum TenantRole

En Java, un enum es un tipo que solo puede tener valores predefinidos. Es la forma idiomática de representar un conjunto fijo de opciones — como roles. 

public enum TenantRole {
    ADMIN,   // puede leer y escribir
    VIEWER   // solo puede leer
}

Paso 2 — El rol viaja dentro del JWT

El rol se guarda en la BD junto al tenant. Cuando hace login, se embebe en el JWT como un claim — un campo extra en el payload del token. Así el servidor sabe el rol sin consultar la BD en cada request. 

// Payload del JWT después de la Fase 12
{
  "sub": "empresa-alpha",   ← tenantId (subject)
  "role": "ADMIN",          ← rol embebido como claim
  "iat": 1712700000,
  "exp": 1712786400
}

Paso 3 — El filtro extrae el rol y lo registra como autoridad

En Spring Security, los permisos se modelan como GrantedAuthority. El filtro JWT ahora extrae el rol del token y lo convierte en una autoridad con el prefijo ROLE_ que Spring espera. 

// JwtAuthenticationFilter — actualizado en Fase 12
String tenantId = jwtUtil.extractTenantId(token);
String role     = "ROLE_" + jwtUtil.extractRole(token).name(); // "ROLE_ADMIN"

var auth = new UsernamePasswordAuthenticationToken(
    tenantId, null,
    List.of(new SimpleGrantedAuthority(role))  // ← aquí va la autoridad
);

Paso 4 — @PreAuthorize protege los endpoints de escritura

Con @EnableMethodSecurity en SecurityConfig, podemos usar @PreAuthorize en cualquier método del controller. Spring evalúa la expresión antes de llamar al método — si falla, lanza 403 automáticamente. 

@PostMapping
@ResponseStatus(HttpStatus.CREATED)
@PreAuthorize("hasRole('ADMIN')")   // ← VIEWER recibe 403 antes de llegar aquí
public ProductResponse create(@AuthenticationPrincipal String tenantId,
                                 @Valid @RequestBody CreateProductRequest request) {
    return productService.createProduct(tenantId, request);
}

401 vs 403 — ¿cuál es la diferencia?

401 Unauthorized: no hay token, o el token es inválido. El cliente no está autenticado.
403 Forbidden: hay token válido, pero el rol no tiene permiso. El cliente está autenticado pero no autorizado.

La diferencia importa: ante un 401 el cliente debería volver a hacer login; ante un 403 debería mostrar "no tenés permisos".
