OmniCore Inventory Engine

Una guía paso a paso de lo que construimos, por qué tomamos cada decisión, y cómo entender el código si es la primera vez que trabajás con Java y Spring Boot. No se asume experiencia previa en Java.

1

¿Qué es este proyecto?

OmniCore Inventory Engine es un sistema backend que gestiona el inventario de productos para múltiples empresas al mismo tiempo. Pensalo como el motor que está detrás de una tienda online: lleva el control de cuántos productos hay en stock, permite agregarlos, modificarlos y consultarlos.

Lo que hace especial a este sistema es que está diseñado para ser usado por muchas empresas distintas a la vez (como un SaaS), manteniendo los datos de cada una completamente separados y seguros.

🏭

Motor de inventario

Gestiona stock de productos en tiempo real con alta concurrencia.

🏢

Multi-empresa (SaaS)

Múltiples clientes usan el mismo sistema, aislados entre sí.

🔒

Seguridad por diseño

La separación de datos es la prioridad número uno.

El stack tecnológico

Si venís de JavaScript/Python, algunos de estos nombres son nuevos. Acá te explicamos qué hace cada uno y por qué lo elegimos.

Tecnología ¿Qué es? ¿Por qué la usamos?
Java 21 El lenguaje de programación. Como Python pero con tipos estrictos. Maduro, rápido, y dominante en backends empresariales.
Spring Boot Un framework para Java. Como Express.js pero para Java. Arranca una app completa con muy poco código de configuración.
PostgreSQL La base de datos. Como MySQL pero más potente. Soporte nativo para concurrencia, tipos avanzados y transacciones robustas.
JPA / Hibernate Una capa que conecta Java con la base de datos. Escribís clases Java y él genera el SQL automáticamente.
Lombok Una librería que genera código repetitivo automáticamente. Evita escribir getters, setters y constructores a mano.
MapStruct Convierte objetos de un tipo a otro automáticamente. Evita copiar campos a mano entre la entidad y el DTO.
Testcontainers Levanta una base de datos real en Docker para los tests. Los tests son 100% fieles a producción. Sin sorpresas.
3

Multi-tenancy: el concepto más importante

Tenant = cliente / empresa. Multi-tenancy significa que el sistema sirve a muchas empresas al mismo tiempo, en una sola instalación.

Imaginá que sos dueño de un edificio con muchos departamentos. Cada familia (tenant) tiene su espacio privado. Comparten el edificio, pero nunca pueden entrar al departamento de otro. Eso es lo que hacemos con los datos.

¿Cómo lo implementamos?

Usamos la estrategia más simple: una sola base de datos, una sola tabla, pero cada fila tiene una columna tenant_id que identifica a qué empresa pertenece. 

-- Tabla products en PostgreSQL
SELECT * FROM products;

 id | tenant_id      | sku      | name          | stock
----+----------------+----------+---------------+-------
  1 | empresa-alpha  | SKU-001  | Widget Pro    |   100  ← empresa A
  2 | empresa-alpha  | SKU-002  | Gadget X      |    50  ← empresa A
  3 | empresa-beta   | SKU-001  | Widget Pro    |   200  ← empresa B (¡mismo SKU, diferente empresa!)
¿Por qué este enfoque?

Alternativas más complejas como una base de datos por tenant o un schema por tenant son más caras de mantener. Con una sola base de datos, una sola migración actualiza a todos los clientes. La seguridad la garantizamos en el código.

Las 3 reglas de oro del aislamiento

1. Toda entidad tiene tenantId

Si falta este campo en una tabla, es un bug crítico de seguridad.

2. Todo query filtra por tenantId

Nunca findAll(). Siempre findAllByTenantId("empresa-alpha"). Si buscás por ID, también: findByTenantIdAndId("empresa-alpha", 1).

3. El tenantId nunca viene del usuario

El servidor lo obtiene del token de autenticación. Un usuario no puede mentir y decir que es de otra empresa.

Regla crítica

Un bug que deje a un cliente ver datos de otro cliente es un incidente de seguridad gravísimo (P0). Esta regla no es negociable en ninguna parte del código.

4

TDD — Tests primero, código después

TDD (Test Driven Development) es una metodología donde primero escribís el test que falla, y después escribís el código mínimo para que ese test pase. Parece al revés, pero tiene mucho sentido.

🔴 Red

Escribís el test. Falla porque el código no existe todavía. Eso está bien.

🟢 Green

Escribís el código mínimo para que el test pase. Sin agregar de más.

🔵 Refactor

Mejorás el código sin romper los tests. Limpieza sin miedo.

¿Por qué trabajar así?

Los tests son la red de seguridad del proyecto. Si en el futuro alguien cambia un método y rompe algo, los tests lo detectan antes de que llegue a producción. También obligan a diseñar código que sea fácil de testear, lo que suele significar código más simple y limpio.

5

Testcontainers — Base de datos real en tests

Cuando corrés los tests, Testcontainers levanta automáticamente un contenedor Docker con PostgreSQL, conecta Spring Boot a él, corre todos los tests, y destruye el contenedor al final. Todo sin que tengas que hacer nada. 

// src/test/java/.../TestcontainersConfiguration.java
@Bean
@ServiceConnection  // ← Spring Boot detecta la URL sola, sin config manual
public PostgreSQLContainer postgresContainer() {
    return new PostgreSQLContainer("postgres:latest");
}

¿Por qué no usamos H2 (base de datos en memoria)?

H2 en memoriaTestcontainers (PostgreSQL real)
Velocidad Muy rápido Algo más lento (levanta Docker)
Fidelidad Simula SQL, con diferencias Idéntico a producción
Sorpresas en producción Frecuentes Ninguna
Tipos nativos de Postgres No soporta (UUID, JSONB…) Funciona igual que en prod
Analogía

Testear con H2 en lugar de PostgreSQL es como ensayar una cirugía en un maniquí de plástico. Testcontainers usa el "paciente real" en un entorno controlado.

Estructura del proyecto

Toda la lógica vive en el paquete com.omnicore.inventory_engine. Dentro, organizamos el código en capas.

src/
  main/java/com/omnicore/inventory_engine/
    InventoryEngineApplication.java  ← punto de entrada de la app
    domain/
      entity/
        Product.java  ← la tabla "products" en Java
      repository/
        ProductRepository.java  ← queries a la base de datos
      service/  ← (próximamente) lógica de negocio
    api/
      controller/  ← (próximamente) endpoints REST
      dto/  ← (próximamente) lo que ve el mundo exterior
      mapper/  ← (próximamente) MapStruct

  test/java/com/omnicore/inventory_engine/
    TestcontainersConfiguration.java  ← levanta PostgreSQL en Docker
    domain/repository/
      ProductRepositoryTest.java  ← 4 tests de persistencia
  test/resources/
    application.properties  ← config solo para tests (create-drop)

¿Por qué separar en domain y api?

domain es el corazón del sistema: las reglas de negocio, la base de datos, la lógica. No sabe nada de HTTP ni de cómo se ve la respuesta.

api es la fachada: lo que el mundo exterior ve. Transforma los datos del dominio en respuestas HTTP limpias. Si mañana queremos agregar una API GraphQL, solo tocamos api/ sin tocar el dominio.

La entidad Product

Una entidad en JPA es una clase Java que representa una tabla en la base de datos. Cada instancia de la clase es una fila en esa tabla. 

@Entity                          // ← "esta clase es una tabla"
@Table(name = "products")         // ← nombre de la tabla en PostgreSQL
@Getter @Setter                  // ← Lombok genera getters y setters
@NoArgsConstructor               // ← Lombok genera constructor vacío (JPA lo necesita)
@AllArgsConstructor @Builder     // ← Lombok genera constructor completo y patrón Builder
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;              // ← la base de datos asigna el ID automáticamente

    @Column(name = "tenant_id", nullable = false)
    private String tenantId;       // ← a qué empresa pertenece este producto

    @Column(nullable = false)
    private String sku;            // ← código único del producto por empresa

    @Column(nullable = false)
    private String name;           // ← nombre del producto

    private String description;    // ← opcional, puede ser null

    @Column(nullable = false)
    private Integer stock;         // ← unidades disponibles

    @Version
    private Long version;          // ← protección contra escrituras simultáneas
}

¿Qué es @Version?

Imaginá que dos usuarios compran el último producto al mismo tiempo. Sin protección, los dos podrían decrementar el stock y quedarte con un número negativo. @Version implementa bloqueo optimista: cada vez que se modifica una fila, el número de versión sube. Si dos personas intentan modificar la misma fila al mismo tiempo, la segunda recibe un error y debe reintentar. Esto es esencial en un motor de inventario.

¿Qué es Lombok?

En Java "puro", tendrías que escribir a mano docenas de líneas de código repetitivo (getters, setters, constructores). Lombok las genera automáticamente durante la compilación con solo una anotación. El código es más corto y legible, sin perder funcionalidad.

ProductRepository — Hablar con la base de datos

El repositorio es la única parte del código que hace queries a la base de datos. Con Spring Data JPA, en muchos casos solo definís la firma del método y Spring genera el SQL automáticamente. 

public interface ProductRepository extends JpaRepository<Product, Long> {

    // Spring genera: SELECT * FROM products WHERE tenant_id = ?
    List<Product> findAllByTenantId(String tenantId);

    // Spring genera: SELECT * FROM products WHERE tenant_id = ? AND sku = ?
    Optional<Product> findByTenantIdAndSku(String tenantId, String sku);

    // Spring genera: SELECT * FROM products WHERE tenant_id = ? AND id = ?
    // Protege el GET por ID: aunque alguien adivine un ID ajeno, retorna vacío
    Optional<Product> findByTenantIdAndId(String tenantId, Long id);
}
¿Por qué usamos Optional?

Optional<Product> es como decirle a quien llama al método: "puede que encuentre el producto, o puede que no". Obliga al código que lo usa a manejar el caso donde no existe, evitando el clásico error NullPointerException.

Regla de aislamiento

Notá que ningún método permite consultar sin tenantId. Si alguien quisiera agregar un findBySku(String sku), devolvería resultados de todos los tenants — eso está prohibido.

9

Los tests — Qué validamos y por qué

Tenemos 4 tests en ProductRepositoryTest. Cada uno valida una responsabilidad distinta.

TEST 1   shouldPersistProductWithAllFields

Guarda un producto y verifica que la base de datos le asignó un ID y una versión. Confirma que el mapeo entre la clase Java y la tabla SQL es correcto.

TEST 2   shouldIsolateDataBetweenTenants

Crea productos de dos empresas distintas. Confirma que al buscar los de la empresa A, no aparecen los de la empresa B. Este es el test de seguridad más importante.

TEST 3   shouldFindProductByTenantIdAndSku

Verifica que podés encontrar un producto específico usando la combinación empresa + código de producto (SKU).

TEST 4   shouldNotFindProductByIdFromDifferentTenant

Escenario de ataque: la empresa B intenta acceder a un registro usando el ID de un producto de la empresa A. El resultado debe ser vacío. Si este test falla, hay una brecha de seguridad.

Para correr los tests

Necesitás Docker corriendo. Luego: ./mvnw test -Dtest=ProductRepositoryTest
